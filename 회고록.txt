# PickPickBoom 회고록

## 1. UI 제작 회고

이번 UI 작업에서 가장 오래 걸린 부분은 정확한 수치를 몰랐는데도, 처음부터 완성 위치를 바로 맞추려고 한 점이었다.

실제로 필요했던 것은 "정답 수치"가 아니라 "나중에 Play Mode에서 빠르게 튜닝할 수 있는 구조"였다.

처음부터 아래처럼 요청했으면 훨씬 빨리 도달할 수 있었다.

```text
Unity 카드 보드 UI를 만들고 싶다.
정확한 위치와 크기 수치는 아직 모른다.
처음부터 완성 수치를 맞추려 하지 말고, Play Mode에서 Inspector 값으로 빠르게 조절 가능한 구조로 만들어줘.

Board Scroll View 구조는 이렇게 해줘.

Viewport
  CardContent
  Ground

CardContent는 카드 행들을 담는다.
각 행은 wall 배경과 카드 3장을 가진다.
Ground는 Viewport의 자식이고, CardContent 다음 sibling에 둔다.

Ground는 화면 하단에 고정되는 이미지가 아니다.
프리뷰 중에는 가장 아래 wall의 아랫변에 붙어서 wall과 함께 움직이는 것처럼 보여야 한다.

Inspector에서 조절할 값:
- cardContentOffset
- cardWidthScale
- cardHeightScale
- rowFocusOffsetY
- previewPanEndOffsetY
- wallTileScale
- wallTileWidthScale
- wallTileSpacing
- groundWallGap
- groundHeightMultiplier

프리뷰 이동량은 0~1 normalized 값으로 조절하지 말고,
UI 좌표 기준 절대 offset으로 조절하게 해줘.

카드 선택 후 실제 행 이동 위치는 rowFocusOffsetY로 조절하게 해줘.
층 수가 늘어나도 같은 느낌으로 보여야 한다.

내가 Inspector에서 값을 맞춘 뒤, 그 값을 코드 기본값으로 저장할 수 있게 해줘.
```

제작 전에는 최종 UI 수치를 알 수 없다.
따라서 프롬프트에는 결과론적인 숫자를 적는 대신, 개발자가 직접 보면서 가장 빠르게 조절할 수 있는 튜닝 방법을 요구해야 한다.

가장 빠른 방식은 다음과 같다.

```text
보드 UI의 정확한 수치는 아직 알 수 없다.
그러므로 모든 배치 수치는 Inspector에서 조절 가능한 튜닝 필드로 만들어줘.

Play Mode 중에도 값이 즉시 반영되게 해줘.
값을 바꾸면 카드 크기, 카드 위치, wall 크기, ground 위치, 프리뷰 종료 위치가 바로 갱신되어야 한다.

Scene View와 Game View를 보면서 다음 순서로 조절할 수 있게 해줘.

1. 카드 크기 조절
   - cardWidthScale
   - cardHeightScale

2. 카드 전체 위치 조절
   - cardContentOffset
   - bottomAlignedCardOffset

3. 실제 선택 행의 화면 위치 조절
   - rowFocusOffsetY

4. 프리뷰가 어디까지 내려갈지 조절
   - previewPanEndOffsetY
   - 0~1 normalized 값이 아니라 UI 좌표 기준 절대 offset으로 조절

5. wall 배경 크기와 간격 조절
   - wallTileScale
   - wallTileWidthScale
   - wallTileVisualHeightScale
   - wallTileHeightMultiplier
   - wallTileSpacing

6. ground 위치와 wall 사이 간격 조절
   - groundOffset
   - groundWallGap
   - groundHeightMultiplier

튜닝이 끝나면 내가 맞춘 Inspector 값을 알려줄 테니,
그때 그 값을 코드 기본값으로 반영해줘.
```

이 방식의 장점은 처음부터 정답 수치를 맞히려고 시간을 쓰지 않는다는 것이다.
먼저 구조를 만들고, 그 다음 실제 화면을 보며 값을 조절하고, 마지막에 확정값만 코드 기본값으로 저장하면 된다.

UI 작업에서 중요한 교훈은 다음과 같다.

- 수치를 모르면 "맞춰줘"가 아니라 "조절 가능하게 만들어줘"라고 요청해야 한다.
- Hierarchy 구조를 먼저 지정해야 한다.
- Anchor, Pivot, Sibling 순서를 명확히 말해야 한다.
- 움직이는 UI는 "화면 기준"인지 "다른 UI 오브젝트 기준"인지 먼저 정해야 한다.
- ScrollRect에서는 0~1 normalized 값보다 절대 UI offset이 튜닝하기 쉽다.
- Ground처럼 특정 오브젝트를 따라가야 하는 경우, 수동 스크롤 오프셋보다 실제 기준 오브젝트의 위치를 읽어 붙이는 방식이 안정적이다.

## 2. 게임 전체 기획 회고

이 게임은 3장의 카드 중 하나를 고르며 탑을 올라가는 카드 선택 로그라이트 게임이다.

핵심 재미는 "짧은 기억", "위험 감수", "운", "선택 후 결과 확인"에서 나온다.

플레이어는 한 층이 시작될 때 전체 보드를 잠깐 미리 본다.
그 뒤 모든 카드가 뒤집히고, 가장 아래 행부터 한 줄씩 선택한다.
각 행에는 카드가 3장 있고, 플레이어는 현재 행의 카드 중 하나만 선택할 수 있다.

선택 결과는 다음과 같다.

- 계단: 다음 행 또는 다음 층으로 진행한다.
- 보물 상자: 현재 층을 통과하거나 좋은 결과를 준다.
- 몬스터: 방패가 없으면 게임 오버, 방패가 있으면 방패를 소모하고 생존한다.
- 방패: 몬스터를 한 번 막아주는 보험을 얻는다.
- 저주: 이전 행으로 되돌아가거나 진행을 방해한다.
- 빈 카드: 아무 일도 없고 같은 행에서 다시 선택한다.

층이 올라갈수록 행 수가 늘어나고, 미리 봐야 하는 정보가 많아진다.
초반에는 기억하기 쉽지만, 후반으로 갈수록 위험 카드와 안전 카드 위치를 기억하기 어려워진다.
따라서 플레이어는 "방금 본 정보를 얼마나 기억하는가", "위험해 보여도 계단일 가능성이 있는 카드를 고를 것인가", "방패를 얻기 위해 같은 행에서 더 버틸 것인가"를 판단한다.

게임 진행 흐름은 다음과 같다.

1. 난이도를 선택한다.
2. 1층 보드가 생성된다.
3. 전체 카드가 잠깐 공개된다.
4. 보드가 위에서 아래로 프리뷰된다.
5. 모든 카드가 뒤집힌다.
6. 가장 아래 행부터 카드 1장을 선택한다.
7. 카드 결과를 적용한다.
8. 진행 성공 시 다음 행 또는 다음 층으로 이동한다.
9. 몬스터에게 당하면 게임 오버가 된다.
10. 도달한 최고 층을 난이도별로 저장한다.

난이도는 다음 요소를 조절한다.

- 보드 프리뷰 속도
- 카드 공개 유지 시간
- 카드 뒤집기/공개 애니메이션 속도
- 빈 카드, 저주, 좋은 아이템 등장 가중치

기획 단계에서 처음부터 정확한 수치를 요구할 필요는 없다.
대신 다음 항목을 조절 가능하게 요청해야 한다.

- 행 수 증가 방식
- 카드 종류별 등장 확률
- 프리뷰 속도
- 카드 공개 유지 시간
- 행 이동 속도
- 몬스터/방패/저주 효과
- UI 위치와 크기
- 보드 배경과 Ground의 기준 위치

## 3. 처음부터 제공했으면 좋았을 전체 게임 제작 프롬프트

아래 프롬프트는 비슷한 게임을 처음부터 제작할 때 바로 사용할 수 있는 형태다.

```text
Unity 2D UI 기반 카드 선택 로그라이트 게임을 만들고 싶다.
항상 한국어로 설명하고, README나 문서도 한국어로 작성해줘.
입력 시스템은 반드시 Unity New Input System만 사용해줘.
StandaloneInputModule이나 구 Input Manager 기반 코드는 사용하지 마.

게임 제목은 PickPickBoom이다.

게임 핵심은 3장의 카드 중 하나를 고르며 탑을 올라가는 짧은 기억 기반 카드 선택 게임이다.
플레이어는 매 층 시작 시 전체 보드를 잠깐 보고, 카드 위치를 기억한 뒤, 가장 아래 행부터 한 줄씩 카드를 선택한다.

재미의 중심은 다음과 같다.
- 짧은 시간 동안 공개된 카드 위치를 기억하는 재미
- 계단, 몬스터, 아이템, 저주 중 무엇을 고를지 고민하는 위험 감수
- 방패를 얻었을 때 몬스터를 한 번 버티는 안도감
- 층이 올라갈수록 행 수가 늘어나 기억 부담이 커지는 긴장감
- 최고 층 기록을 갱신하는 반복 플레이 동기

기본 규칙:
- 한 행에는 카드 3장이 있다.
- 플레이어는 현재 행의 카드 3장 중 하나만 선택할 수 있다.
- 층이 올라갈수록 행 수가 늘어난다.
- 매 층 시작 시 전체 카드가 먼저 공개된다.
- 보드가 위에서 아래로 프리뷰된다.
- 프리뷰가 끝나면 모든 카드가 뒤집힌다.
- 가장 아래 행부터 선택을 시작한다.
- 성공적으로 진행하면 다음 행으로 이동한다.
- 최상단까지 통과하면 다음 층이 생성된다.
- 몬스터를 방패 없이 고르면 게임 오버다.
- 최고 도달 층을 난이도별로 저장한다.

카드 종류:
- Stair: 다음 행 또는 다음 층으로 진행
- Chest: 층 통과 또는 보상 카드
- Monster: 방패가 없으면 게임 오버, 방패가 있으면 방패 소모 후 생존
- Shield: 방패 1개 획득, 방패는 최대 1개만 보유
- Curse: 진행을 방해한다. 기본 구현은 2행 전으로 되돌아가고 해당 구간 카드를 다시 뒤집는 방식으로 한다.
- Empty: 아무 일도 없음, 같은 행에서 다시 선택

층 생성:
- 각 일반 행은 Stair 1장, Monster 1장, 랜덤 카드 1장으로 구성한다.
- 랜덤 카드는 Empty, Curse, Shield 중 가중치로 선택한다.
- 각 행의 카드 3장은 무작위로 섞는다.
- 마지막 또는 특수 행에는 Chest를 배치할 수 있게 구조를 열어둔다.
- FloorGenerator가 층 번호를 받아 전체 카드 리스트를 생성하게 한다.

난이도:
- Easy, Normal, Hard를 만든다.
- 난이도별로 다음 값을 조절 가능하게 한다.
  - 보드 프리뷰 행당 시간
  - 카드 공개 유지 시간
  - 카드 공개 애니메이션 시간
  - 결과 적용 전 딜레이
  - 층 전환 딜레이
  - Empty / Curse / Shield 가중치

Board Scroll View 요구사항:
- CardContent는 Viewport 안에 둔다.
- CardContent는 런타임에 BoardRow들을 가진다.
- 각 BoardRow는 wall 배경과 카드 3장을 가진다.
- Ground는 Viewport의 자식이며 CardContent 다음 sibling에 둔다.
- Ground는 화면 하단 고정이 아니라 가장 아래 wall의 아랫변에 붙어서 움직이게 한다.
- 프리뷰 중 가장 아래 wall이 올라오면 Ground도 같이 붙어서 올라오는 것처럼 보여야 한다.
- wall과 ground 사이 간격은 groundWallGap으로 Inspector에서 조절한다.

UI 튜닝 값:
- 정확한 수치는 처음에는 모른다.
- 아래 값을 Inspector에서 조절 가능하게 만들고 Play Mode 중 변경하면 즉시 반영되게 해줘.
  - cardContentOffset
  - cardContentScale
  - cardWidthScale
  - cardHeightScale
  - rowFocusOffsetY
  - previewPanEndOffsetY
  - wallTileOffset
  - wallTileScale
  - wallTileWidthScale
  - wallTileVisualHeightScale
  - wallTileHeightMultiplier
  - wallTileSpacing
  - groundOffset
  - groundWallGap
  - groundHeightMultiplier

프리뷰 이동:
- 0~1 normalized 값으로 조절하지 않는다.
- UI 좌표 기준 절대 offset으로 조절한다.
- 층 수가 늘어나도 같은 느낌으로 보이게, 현재 층의 마지막 행 위치를 기준으로 previewPanEndOffsetY를 더해 계산한다.

카드 선택 후 이동:
- 실제 선택 행 위치는 rowFocusOffsetY로 조절한다.
- rowFocusOffsetY도 절대 UI 좌표 기준으로 둔다.
- 카드 선택 후 다음 행으로 이동할 때 ScrollRect를 부드럽게 이동시킨다.

개발 방식:
- 정확한 배치 수치를 처음부터 강제로 정하지 않는다.
- 먼저 작동하는 구조와 Inspector 튜닝 필드를 만든다.
- 내가 Play Mode에서 값을 맞춘 뒤, 그 값을 코드 기본값으로 반영할 수 있게 해줘.
- 코드는 기존 Unity UI 패턴에 맞춰 작성하고, 불필요한 런타임 UI 자동 생성은 최소화한다.
- 단, Ground와 BoardRow처럼 층 수에 따라 필요한 요소는 런타임 생성해도 된다.

검증:
- C# 컴파일이 통과해야 한다.
- New Input System 기준으로 버튼 클릭이 동작해야 한다.
- 1층뿐 아니라 2층, 3층 이상에서도 프리뷰가 어색하지 않아야 한다.
- Ground가 갑자기 화면 하단에 튀어나오지 않아야 한다.
- 카드 선택 후 현재 행만 클릭 가능해야 한다.
- 게임 오버 후 재시작과 메인 메뉴 이동이 가능해야 한다.
```

## 4. 다음 프로젝트에서 더 빠르게 가기 위한 요청 방식

앞으로 비슷한 게임을 만들 때는 다음 순서로 요청하는 것이 좋다.

```text
1. 먼저 게임의 재미와 진행 규칙을 정리해줘.
2. 그 규칙을 기준으로 필요한 스크립트 구조를 제안해줘.
3. 정확한 UI 수치는 모르니, Inspector 튜닝 필드를 먼저 설계해줘.
4. UI Hierarchy, Anchor, Pivot, Sibling 순서를 명시해줘.
5. 첫 구현은 완성 수치가 아니라 튜닝 가능한 프로토타입으로 만들어줘.
6. 내가 Play Mode에서 맞춘 값을 알려주면 코드 기본값으로 저장해줘.
7. 마지막으로 전체 제작 프롬프트를 회고록에 정리해줘.
```

즉, 처음부터 "예쁘게 배치해줘"라고 요청하기보다,
"어떤 기준으로 배치되고, 어떤 값들을 나중에 조절할 수 있어야 하는지"를 먼저 요청하는 것이 가장 빠르다.
